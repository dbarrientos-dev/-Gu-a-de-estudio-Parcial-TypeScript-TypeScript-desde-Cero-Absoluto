Instalación en Fedora con Bun

El profe usa Bun como runtime y gestor de paquetes. Bun reemplaza a Node.js para este curso porque puede ejecutar archivos .ts directamente sin compilarlos primero. Es como si Bun hablara TypeScript de forma nativa.

💡
¿Por qué Bun y no Node.js?

Node.js no puede ejecutar archivos .ts directamente. Necesitas herramientas extra (ts-node, esbuild, etc.). Bun los ejecuta nativamente, lo que simplifica mucho el flujo de desarrollo.

01
// conceptos base

¿Qué es TypeScript?

TypeScript es JavaScript con tipos. Es un superset de JavaScript: todo código JavaScript válido también es TypeScript válido. La diferencia es que TypeScript te obliga a declarar qué tipo de dato puede tener cada variable.

TypeScript (.ts) → compila a → JavaScript (.js) → ejecuta en → Bun / Node / Navegador
Con Bun: el paso de compilación es automático e invisible ✓

¿Por qué sirve?

En JavaScript puro esto es válido y no da error hasta que ya estás corriendo el programa:

javascriptJavaScript — sin seguridad
let nombre = "Daniel";
nombre = 42;          // JavaScript: OK (sin error)
nombre.toUpperCase(); // 💥 Error en EJECUCIÓN: number no tiene toUpperCase

En TypeScript, el error aparece antes de ejecutar, mientras escribes código:

typescriptTypeScript — error en compilación ✓
let nombre = "Daniel";
nombre = 42;
// ❌ Error: Type 'number' is not assignable to type 'string'
// El error aparece en el editor, ANTES de correr el código
✅
Regla de oro de TypeScript

TypeScript detecta errores en tiempo de compilación (cuando escribes / guardas el código). JavaScript los detecta en tiempo de ejecución (cuando ya está corriendo, potencialmente frente a usuarios).

Los tipos primitivos son los bloques básicos. Aquí están todos los que debes conocer:

TipoDescripciónEjemplo

numberEnteros y decimales42, 3.14
booleanVerdadero o falsotrue, false

typescriptDeclaración correcta de tipos primitivos
// Sintaxis: let nombreVariable: tipo = valor;

let nombre: string = "María";       // ✅ texto
let edad: number = 25;              // ✅ número entero
let activo: boolean = true;         // ✅ verdadero/falso

// bigint: para números GIGANTES que superan la precisión de number
// IMPORTANTE: el sufijo "n" al final del número es OBLIGATORIO
let distancia: bigint = 9461000000000000n;  // ✅ un año luz en metros

// ─── Errores comunes ───
let nombre2: string = 25;           // ❌ Error: number ≠ string
let activo2: boolean = "true";      // ❌ Error: string ≠ boolean
let distancia2: bigint = 9461000000000000; // ❌ Falta la "n" → no es bigint
⚠️
¡Trampa del parcial con bigint!

Sin el sufijo n, el número es un number normal que pierde precisión. 9461000000000000n es bigint. 9461000000000000 es number (inexacto). El tipo boolean solo acepta true o false, NUNCA el string "true".

03
// type inference

Inferencia de Tipos

TypeScript es inteligente. Si declaras una variable con un valor, infiere automáticamente el tipo sin que tú lo escribas. Esto es la inferencia de tipos.

typescriptInferencia automática
// TypeScript INFIERE el tipo según el valor inicial:
let ciudad = "Bogotá";    // TypeScript infiere: string
let temperatura = 25;      // TypeScript infiere: number
let esLluvioso = true;     // TypeScript infiere: boolean

// Una vez inferido el tipo, NO puede cambiarse:
ciudad = "Medellín";  // ✅ string → string, OK
ciudad = 42;          // ❌ Error: Type 'number' is not assignable to type 'string'

// La inferencia también funciona con template literals:
let saludo = `Hola, ${ciudad}`;  // TypeScript infiere: string — OK
🧠
Inferencia vs Anotación explícita

let x = "hola" → inferencia (TS adivina que es string).
let x: string = "hola" → anotación explícita (tú le dices el tipo).
Ambas son válidas. La anotación explícita es más clara en proyectos grandes.

¿Qué fragmento tiene error?

✓ Sin error
let mensaje = "Hola";
mensaje = "Adiós";  // string → string ✓

let total = 100;
total = 200;        // number → number ✓

let esValido = true;
esValido = false;   // boolean → boolean ✓
✗ Error de compilación
let ciudad = "Bogotá";
// TypeScript infiere: string

ciudad = 42;
// ❌ Error: Type 'number' is not
// assignable to type 'string'
04
// tipos especiales

any vs unknown vs never

Estos tres tipos especiales controlan cómo TypeScript maneja valores cuyo tipo no es conocido o que nunca deberían ocurrir.

any — El tipo "rendirse" 😞

Con any le dices a TypeScript: "no revises nada aquí". Desactiva toda la seguridad de tipos. En 2026 es una mala práctica grave.

typescriptany — mala práctica
function procesar(dato: any): string {
  return dato.toUpperCase();  // ✅ TypeScript no objeta...
  // ❌ Pero si "dato" es un número, EXPLOTA en ejecución
  // any = seguridad cero
}

procesar(42);       // No da error de compilación
procesar("hola");   // Tampoco
// Los dos compilan, pero el primero FALLA al ejecutar

unknown — El tipo "no sé, pero verifica" ✅

unknown también acepta cualquier valor, PERO antes de usar el valor, TypeScript te exige verificar qué tipo es. Es la versión segura de any.

typescriptunknown + type guards — la forma correcta
function procesar(dato: unknown): string {
  // ANTES de usar "dato", debemos verificar su tipo:
  if (typeof dato === "string") {
    // Dentro de este bloque, TypeScript SABE que dato es string
    return dato.toUpperCase();    // ✅ seguro
  }
  if (typeof dato === "number") {
    // Aquí TypeScript sabe que dato es number
    return dato.toFixed(2);       // ✅ seguro
  }
  return "Tipo no soportado";
}

// Esto se llama "type guard" — verificar tipo con typeof antes de usar

never — El tipo "esto no puede pasar"

never representa valores que nunca deberían existir. Útil para el patrón exhaustive checking (explicado en la sección 11).

typescriptnever — casos imposibles
// Una función que NUNCA retorna (bucle infinito o siempre lanza error):
function error(mensaje: string): never {
  throw new Error(mensaje);  // Nunca llega al return
}

// never como tipo de variable: no se puede asignar ningún valor
let x: never;
// x = "algo";  // ❌ Error — ningún valor satisface never
Tipo¿Acepta?¿Requiere verificación?Usar cuando...
anyTodoNoNunca (mala práctica)
unknownTodoSí (type guards)Datos de APIs externas, JSON desconocido
neverNadaN/AFunciones que nunca retornan, exhaustive checks
🚫
En examen: any = respuesta incorrecta casi siempre

Si una opción usa any, descártala. El profe busca código seguro. La respuesta correcta usa unknown con type guards.

Arrays y Tuplas

Arrays — listas de un tipo

typescriptArrays tipados
// Forma 1: tipo[]
let numeros: number[] = [1, 2, 3, 4, 5];
let nombres: string[] = ["Ana", "Luis", "Sara"];

// Forma 2: Array  (equivalente)
let numeros2: Array = [1, 2, 3];

// Los arrays son dinámicos: pueden crecer con push, pop, etc.
nombres.push("Carlos");  // ✅ OK — agrega al final
nombres.push(42);        // ❌ Error: number no es string

// Array de sedes (dinámico):
let sedes: string[] = ["Norte", "Sur"];
sedes.push("Centro");   // ✅ puede crecer sin límite

Tuplas — listas con estructura fija

Las tuplas tienen longitud fija y tipos definidos por posición. La posición 0 es de un tipo, la posición 1 de otro, etc.

typescriptTuplas
// Tupla simple: exactamente [number, string]
let coordenada: [number, string] = [4.60, "Norte"];

// Tupla con etiquetas (más descriptivo):
let punto: [latitud: number, longitud: number] = [4.60, -74.08];

// readonly: hace la tupla INMUTABLE (no se puede modificar)
const coordInmutable: readonly [latitud: number, longitud: number] = [4.60, -74.08];
// coordInmutable[0] = 5;  // ❌ Error: cannot assign to readonly

// ─── La respuesta correcta del parcial (Pregunta 5) ───
const coordenada2: readonly [latitud: number, longitud: number] = [4.60, -74.08];
let sedes: string[] = ["Norte", "Sur"];
sedes.push("Centro");  // ✅ el array puede crecer, la tupla no
number[] — Array normal
let arr: number[] = [4.60, -74.08];
// ❌ NO es tupla:
// - No fija la longitud
// - Acepta más o menos elementos
// - Sin nombres descriptivos
arr.push(100); // Permite
readonly [number, number] — Tupla
const coord: readonly [latitud: number, longitud: number]
  = [4.60, -74.08];
// ✅ ES tupla:
// - Exactamente 2 elementos
// - Posición 0 = latitud (number)
// - Posición 1 = longitud (number)
// - readonly = no se puede modificar

// union types · literal types · type aliases

Union Types y Type Aliases

Union Types — "puede ser A o B"

El operador | crea tipos que pueden ser uno u otro.

typescriptUnion Types
// Un ID puede ser UUID (string) o número legacy
type ID = string | number;

let userId: ID = "abc-123";  // ✅ string
userId = 456;                 // ✅ number también es válido
userId = true;                // ❌ Error: boolean no está en la unión

// ─── Intersection vs Union ───
type Intersection = string & number; // = never (imposible: nada es string Y number al mismo tiempo)
type Union = string | number;        // OK: puede ser uno O el otro

Literal Types — valores exactos

Puedes restringir un tipo a valores específicos, no solo al tipo general:

typescriptLiteral Types
// Sin literal types: acepta CUALQUIER string
let estado: string = "casado";
estado = "zombi";  // ✅ TS lo permite, aunque no tiene sentido

// CON literal types: solo acepta esos valores exactos
type EstadoCivil = "soltero" | "casado" | "divorciado";
let mi_estado: EstadoCivil = "soltero";   // ✅
mi_estado = "casado";                      // ✅
mi_estado = "zombi";                       // ❌ Error: no está en la unión

Type Aliases — nombres para tipos

typescriptType Aliases completo
// "type" crea un ALIAS — un nombre para un tipo
type ID = string | number;
type EstadoCivil = "soltero" | "casado" | "divorciado";

// Uso:
let userId: ID = "abc-123";
userId = 456;  // Válido — ambos están en la unión

let estado: EstadoCivil = "soltero";
// estado = "separado"; // ❌ no está en el alias

// ─── interface NO puede hacer esto ───
// interface ID = string | number;  // ❌ SINTAXIS INVÁLIDA
// Las interfaces son SOLO para objetos, no para uniones primitivas
⚠️
interface vs type — diferencia clave

interface solo sirve para describir la forma de objetos. type sirve para todo: uniones, intersecciones, tipos condicionales, alias de primitivos. Si necesitas string | number, usa type.

// optional chaining · nullish coalescing

Operadores ?. y ??

?. — Optional Chaining (acceso seguro)

Si intentas acceder a una propiedad de algo que podría ser null o undefined, tu programa explota. ?. lo evita: si el objeto no existe, devuelve undefined en lugar de lanzar error.

typescriptOptional Chaining ?.
interface Usuario {
  nombre: string;
  direccion?: {       // La ? significa que es opcional (puede no existir)
    calle: string;
    piso?: number;
  };
}

const usuario: Usuario = { nombre: "Daniel" }; // Sin dirección

// SIN ?. — EXPLOTA si dirección no existe:
// const calle = usuario.direccion.calle;  // ❌ TypeError en ejecución

// CON ?. — Seguro:
const calle = usuario.direccion?.calle;  // ✅ Devuelve undefined (no explota)
const piso = usuario.direccion?.piso;    // ✅ También undefined

?? — Nullish Coalescing (valor por defecto)

?? proporciona un valor por defecto solo si el lado izquierdo es null o undefined. Ignora otros valores "falsy" como 0, "", false.

typescriptNullish Coalescing ?? vs OR ||
// ?? solo actúa ante null/undefined:
const piso = usuario.direccion?.piso ?? 0;
// Si piso es 0 → devuelve 0  ← ✅ correcto (0 es válido)
// Si piso es undefined → devuelve 0 (el default)
// Si piso es null → devuelve 0 (el default)

// || actúa ante TODOS los valores falsy (0, false, ""):
const piso2 = usuario.direccion?.piso || 1;
// Si piso es 0 → devuelve 1  ← ❌ incorrecto (perdemos el 0)
// Si piso es undefined → devuelve 1

// ─── Combinando ?. y ?? ───
const calle = usuario.direccion?.calle ?? "No registrada";
// Si dirección no existe: undefined ?? "No registrada" = "No registrada"
// Si dirección existe pero calle está vacía: "" ?? "No registrada" = "" (preserva)
OperadorActiva conNo activa con
??null, undefined0, "", false
||Todo lo "falsy": null, undefined, 0, "", falseSolo valores "truthy"
💡
El ! es peligroso

usuario.direccion!.calle — El ! (non-null assertion) le dice a TypeScript "confía en mí, esto existe". Pero si no existe, explota en ejecución. Úsalo solo cuando estés 100% seguro. En el parcial, es casi siempre respuesta incorrecta.

Funciones — Parámetros y Retorno

En TypeScript, las funciones tienen tipos en sus parámetros Y en su valor de retorno.

typescriptAnatomía de una función tipada
//         parámetro obligatorio   parámetro con default  retorno
//                  ↓                        ↓               ↓
function calcularConIVA(precio: number, iva: number = 0.19): number {
  if (precio < 0) throw new Error("Precio inválido");
  return precio * (1 + iva);
}

// Uso:
calcularConIVA(100);         // usa iva = 0.19 (default)
calcularConIVA(100, 0.16);   // usa iva = 0.16 explícito

Parámetros opcionales vs con default

✗ Parámetro opcional (peligroso)
function calc(precio: number, iva?: number): number {
  // iva es opcional → puede ser undefined
  return precio * (1 + iva);
  // ❌ 1 + undefined = NaN !
}
✓ Parámetro con default (correcto)
function calc(precio: number, iva: number = 0.19): number {
  // iva tiene valor por defecto
  return precio * (1 + iva);
  // ✅ Si no se pasa, usa 0.19
}
⚠️
Incompatibilidad de retorno

): string { return numero * iva } — Si declaras retorno string pero la expresión produce number, TypeScript marca error. El tipo declarado debe coincidir con lo que realmente retorna la función.

09
// arrow functions · this léxico

Arrow Functions y el problema de this

Las funciones flecha (=>) tienen una propiedad especial: heredan el this del contexto donde fueron definidas (léxico). Las funciones tradicionales crean su propio this.

✗ function() — this incorrecto
class Temporizador {
  segundos: number = 0;
  iniciar() {
    setInterval(function() {
      this.segundos++;
      // ❌ "this" aquí NO apunta a
      // la instancia de Temporizador.
      // Apunta al contexto global
      // (o undefined en strict mode)
    }, 1000);
  }
}
✓ Arrow function — this correcto
class Temporizador {
  segundos: number = 0;
  iniciar() {
    setInterval(() => {
      this.segundos++;
      // ✅ "this" hereda del método
      // iniciar() → apunta a la
      // instancia de Temporizador ✓
    }, 1000);
  }
}

Arrow functions con .map()

typescriptSintaxis concisa con map
// Función tradicional (verbosa):
const conIVA1 = [100, 200, 300].map(function(p) {
  return p * 1.19;
});

// Arrow function (concisa) — la forma preferida:
const conIVA2 = [100, 200, 300].map((p) => p * 1.19);
// Si el body es una sola expresión, no necesita {} ni return

// ─── Trampa: retornar objetos ───
// ❌ Mal: las {} se interpretan como bloque de código:
const crear1 = (nombre: string) => { nombre, activo: true };

// ✅ Bien: envolver el objeto en paréntesis:
const crear2 = (nombre: string) => ({ nombre, activo: true });

// variadic functions

Parámetros Rest (...)

Los parámetros rest (...) capturan un número indefinido de argumentos en un array tipado. Siempre van al final de la firma de la función.

typescriptRest parameters
// "...numeros" captura todos los argumentos como number[]
function sumar(...numeros: number[]): number {
  return numeros.reduce((total, n) => total + n, 0);
}

sumar(1, 2, 3);        // 6
sumar(10, 20, 30, 40); // 100
sumar();               // 0 (array vacío)

// Combinando parámetro fijo + rest:
function registrar(accion: string, ...detalles: string[]): void {
  console.log(`[${accion}] ${detalles.join(", ")}`);
}

registrar("LOGIN", "usuario:daniel", "ip:192.168.1.1");
// → "[LOGIN] usuario:daniel, ip:192.168.1.1"

// ─── Errores comunes ───
// ❌ El rest NO puede ir antes de un parámetro fijo:
// function mal(...items: string[], titulo: string) {}  // Error de sintaxis
⚠️
Rest vs arguments (legacy)

El objeto arguments es de JavaScript antiguo: no tiene tipos, no funciona en arrow functions. Siempre usa ...params: tipo[] en TypeScript.

// exhaustive checking · never pattern

Switch con Exhaustive Checking

El patrón exhaustive checking usa never en el default del switch para que TypeScript te obligue a manejar todos los casos posibles. Si en el futuro agregas un nuevo valor al tipo, el compilador te avisa.

typescriptEl patrón exhaustive checking
type Semaforo = "rojo" | "amarillo" | "verde";

function accion(color: Semaforo): string {
  switch (color) {
    case "rojo":     return "Detenerse";
    case "amarillo": return "Precaución";
    case "verde":    return "Avanzar";
    default:
      // Si todos los casos están cubiertos, "color" tiene tipo never
      // (es imposible llegar aquí)
      const _exhaustive: never = color;
      return _exhaustive;
      // Si agregas "azul" al tipo Semaforo y olvidas el case "azul",
      // TypeScript marca error aquí: '"azul" no es assignable a never'
  }
}

¿Por qué no es suficiente un default normal?

✗ Default normal — oculta errores
type Semaforo = "rojo" | "amarillo" | "verde";
// Más tarde alguien agrega: | "azul"

switch (color) {
  case "rojo": return "Detenerse";
  case "verde": return "Avanzar";
  // Falta "amarillo" y "azul"
  default: return "Desconocido";
  // ❌ Caen silenciosamente aquí
  // TypeScript NO avisa del caso faltante
}
✓ Never — detecta casos faltantes
type Semaforo = "rojo" | "amarillo" | "verde" | "azul";

switch (color) {
  case "rojo":     return "Detenerse";
  case "amarillo": return "Precaución";
  case "verde":    return "Avanzar";
  // Falta "azul" ↓
  default:
    const x: never = color;
    // ❌ Error: '"azul"' is not assignable
    // to type 'never' → TypeScript AVISA
    return x;
}

// OOP · clases

Clases y Modificadores de Acceso

Las clases son plantillas para crear objetos. TypeScript añade modificadores de acceso que controlan desde dónde se puede acceder a cada propiedad o método.

public
Accesible desde cualquier lugar: dentro de la clase, subclases, y código externo
private
Solo accesible dentro de la propia clase. Ni siquiera subclases pueden accederlo
protected
Accesible dentro de la clase Y en todas sus subclases (pero no externamente)
readonly
Solo puede asignarse en la declaración o en el constructor. Después es inmutable
typescriptCuentaBancaria — modificadores correctos
class CuentaBancaria {
  public titular: string;        // Accesible externamente ✓
  private saldo: number;         // Solo dentro de esta clase
  protected banco: string;       // Accesible en subclases también
  readonly numeroCuenta: string; // Solo lectura después del constructor

  constructor(titular: string, saldo: number, banco: string, numeroCuenta: string) {
    this.titular = titular;
    this.saldo = saldo;
    this.banco = banco;
    this.numeroCuenta = numeroCuenta;
  }

  // Método público para acceder al saldo (con validación):
  depositar(monto: number): void {
    if (monto <= 0) throw new Error("Monto inválido");
    this.saldo += monto;  // ✅ Acceso a private OK desde dentro
  }

  consultarSaldo(): number {
    return this.saldo;  // ✅ OK desde dentro
  }
}

const cuenta = new CuentaBancaria("Daniel", 1000, "Bancolombia", "001-123");

cuenta.titular = "Carlos";     // ✅ public — OK
// cuenta.saldo = 9999;        // ❌ private — Error
// cuenta.banco = "Davivienda"; // ❌ protected — Error (solo subclases)
// cuenta.numeroCuenta = "xxx"; // ❌ readonly — Error
💡
Shorthand en constructores

Puedes poner el modificador directamente en el parámetro del constructor y TypeScript crea automáticamente la propiedad: constructor(private nombre: string) equivale a declarar la propiedad y asignarla en el constructor.

Herencia

Animal
│ (clase padre / base)
extends
Perro
│ (clase hija / derivada)
typescriptHerencia completa — la respuesta del parcial
class Animal {
  constructor(
    public nombre: string,
    public edad: number
  ) {}

  presentarse(): string {
    return `Soy ${this.nombre}, tengo ${this.edad} años`;
  }
}

class Perro extends Animal {    // ← "extends" para herencia de clases
  constructor(
    nombre: string,
    edad: number,
    private raza: string         // ← Propiedad nueva de Perro
  ) {
    super(nombre, edad);         // ← OBLIGATORIO: llama al constructor padre
    // "super()" debe ir ANTES de cualquier uso de "this"
  }

  override presentarse(): string {  // ← "override" explícito (buena práctica)
    // super.presentarse() reutiliza el método del padre:
    return `${super.presentarse()} y soy un ${this.raza}`;
  }
}

Interfaces y Herencia

Las interfaces definen la forma (estructura) que deben tener los objetos. Son contratos: si una clase o objeto dice que implementa/cumple una interfaz, TypeScript verifica que tenga todas las propiedades y métodos requeridos.

typescriptInterfaces básicas
interface Persona {
  nombre: string;
  edad: number;
  email?: string;   // ← La ? hace la propiedad OPCIONAL
}

const daniel: Persona = {
  nombre: "Daniel",
  edad: 20,
  // email es opcional, no hay que incluirlo
};

Herencia entre interfaces (extends)

typescriptExtensión de interfaces
interface EntidadBase {

interface Usuario extends EntidadBase {

Esta es una pregunta frecuente. La regla de oro de 2026:

Genéricos <T>

Los genéricos son como parámetros para los tipos. En lugar de escribir una función para number y otra para string, escribes una sola función que funciona con cualquier tipo que tú especifiques.

typescript¿Por qué necesitamos genéricos?
// ❌ Sin genéricos: hay que escribir una función por tipo:
function primerNumero(arr: number[]): number { return arr[0]; }
function primerString(arr: string[]): string { return arr[0]; }

// ❌ Con "any": funciona, pero perdemos los tipos:
function primero(arr: any[]): any { return arr[0]; }
const x = primero([1, 2, 3]);
// x tiene tipo "any" — TypeScript no sabe que es number

// ✅ Con genéricos: una sola función, tipos preservados:
function primero(arr: T[]): T {
  return arr[0];
}

const numero = primero([1, 2, 3]);    // T = number → resultado: number
const texto = primero(["a", "b"]);    // T = string → resultado: string
// TypeScript sabe el tipo exacto del resultado ✓

Clase genérica: Pila<T> (Stack)

typescriptImplementación correcta de Pila<T>
class Pila {
  private elementos: T[] = [];    // ← array interno usa T, no any

  push(elem: T): void {
    this.elementos.push(elem);
  }

  pop(): T | undefined {          // ← T | undefined: puede estar vacía
    return this.elementos.pop();
  }

  peek(): T | undefined {         // ← ver el tope sin sacarlo
    return this.elementos.at(-1);
  }
}

// Al instanciar, T se concretiza:
const nums = new Pila();
nums.push(10);      // ✅ OK
// nums.push("x");  // ❌ Error: string no es number

const textos = new Pila();
textos.push("hola"); // ✅ OK
🚫
Error de parcial: pop() con non-null assertion (!)

pop(): T { return this.elementos.pop()! } — Si la pila está vacía, pop() devuelve undefined y el ! falla en ejecución. La firma correcta es pop(): T | undefined.

Restricciones en Genéricos

A veces quieres que el genérico solo acepte tipos con ciertas características. T extends SomeThing restringe T para que solo sea válido si cumple con esa estructura.

typescriptT extends ConLongitud — solo tipos con .length
// Definimos qué necesitamos:
interface ConLongitud {

}

// T extends ConLongitud → T DEBE tener la propiedad "length"
function mostrar(elemento: T): string {
  return `Longitud: ${elemento.length}`;
}

mostrar("hola");         // ✅ string tiene .length

// mostrar(true);        // ❌ Error de compilación: boolean no tiene .length

// ─── Sin la restricción: error diferente ───
function mostrarSinRestr(elemento: T): string {

Discriminated Unions

Una discriminated union es una unión donde cada variante tiene una propiedad con un valor literal único que la identifica (el "discriminante"). TypeScript usa ese discriminante para saber exactamente qué propiedades están disponibles en cada caso.

typescriptDiscriminated Union — la forma correcta
// Cada variante tiene "tipo" con un valor LITERAL ÚNICO:
type Figura =
  | { tipo: "circulo"; radio: number }           // discriminante: "circulo"
  | { tipo: "rectangulo"; ancho: number; alto: number };  // discriminante: "rectangulo"

function area(fig: Figura): number {
  switch (fig.tipo) {
    case "circulo":
      // Aquí TypeScript SABE que fig tiene "radio"
      return Math.PI * fig.radio ** 2;    // ✅ acceso seguro

    case "rectangulo":
      // Aquí TypeScript SABE que fig tiene "ancho" y "alto"
      return fig.ancho * fig.alto;        // ✅ acceso seguro
  }
}

area({ tipo: "circulo", radio: 5 });
area({ tipo: "rectangulo", ancho: 4, alto: 6 });
✗ Sin discriminante literal
// tipo: string ← no discrimina
type Figura =
  | { tipo: string; radio: number }
  | { tipo: string; ancho: number; alto: number };

// TypeScript no puede distinguir
// cuál variante es en el switch
// ❌ No hay narrowing
✓ Con discriminante literal
// "circulo" y "rectangulo" son
// valores literales únicos ✓
type Figura =
  | { tipo: "circulo"; radio: number }
  | { tipo: "rectangulo"; ancho: number; alto: number };

// ✅ TypeScript refina el tipo
// en cada case del switch

Operador satisfies

El operador satisfies hace dos cosas a la vez: valida que un objeto cumple un tipo, Y mantiene la inferencia exacta de las propiedades. Es la solución entre anotar el tipo (que pierde precisión) y no anotar nada (que no valida).

typescriptsatisfies vs anotación directa vs as
type Colores = Record;

// ─── Con anotación directa ( :Colores ) ───
const paleta1: Colores = {
  rojo: [255, 0, 0],
  verde: [0, 255, 0],
};
paleta1.amarillo;  // ✅ no da error (la clave es "string")
// Pero TypeScript no sabe qué colores existen realmente

// ─── Con "as" ───
const paleta2 = {
  rojo: [255, 0, 0],
} as Colores;
// No valida que el valor sea correcto
// paleta2.amarillo; // No da error tampoco — pierde inferencia

// ─── Con "satisfies" ✅ ───
const paleta = {
  rojo: [255, 0, 0],
  verde: [0, 255, 0],
  azul: [0, 0, 255],
} satisfies Colores;

paleta.rojo;      // ✅ TypeScript sabe que existe y su tipo exacto
// paleta.amarillo; // ❌ Error: propiedad no existe → detecta errores
// paleta.rojo = [255]; // ❌ Error: no cumple [number, number, number]
Técnica¿Valida la estructura?¿Mantiene inferencia exacta?
const x: Tipo = {...}✅ Sí❌ No (pierde propiedades específicas)
const x = {...} as Tipo❌ No (lo asumes)❌ No
const x = {...} satisfies Tipo✅ Sí✅ Sí — lo mejor de los dos

Async/Await Tipado

El código asíncrono (operaciones que toman tiempo: llamadas a APIs, lectura de archivos, etc.) en TypeScript usa async/await. Las funciones async siempre retornan una Promise.

typescriptFunción asíncrona correctamente tipada
// Definimos la forma de los datos que esperamos de la API:
interface DatosAPI {
  usuarios: { id: number; nombre: string }[];
  total: number;
}

// async → la función retorna Promise automáticamente
// El tipo de retorno es Promise
async function obtenerDatos(url: string): Promise {
  const resp = await fetch(url);          // await pausa hasta que fetch termina

  if (!resp.ok) {                         // Verificación de error HTTP
    throw new Error(`HTTP ${resp.status}`);
  }

  const datos: DatosAPI = await resp.json();  // Tipamos el JSON
  return datos;
}

// Uso:
async function main() {
  const datos = await obtenerDatos("https://api.ejemplo.com/usuarios");
  console.log(datos.total);          // TypeScript sabe que es number ✓
  console.log(datos.usuarios[0].nombre); // TypeScript sabe que es string ✓
}
🚫
Error frecuente en parcial

async function f(): DatosAPI — Incorrecto. Las funciones async siempre retornan Promise. Debe ser async function f(): Promise<DatosAPI>.

Patrón Result

El patrón Result es una alternativa a las excepciones (try/catch). En lugar de lanzar un error, la función retorna un objeto que puede ser éxito o fallo, y TypeScript sabe exactamente qué propiedades existen en cada caso.

✗ Con excepciones
function dividir(a: number, b: number): number {
  if (b === 0) throw new Error("División por cero");
  return a / b;
}

try {
  const r = dividir(10, 0);
} catch (e) {
  // ❌ "e" es de tipo unknown
  // No hay certeza del tipo del error
}
✓ Con patrón Result
type Result =
  | { success: true; data: T }
  | { success: false; error: E };

function dividir(a: number, b: number): Result {
  if (b === 0) return { success: false, error: "División por cero" };
  return { success: true, data: a / b };
}

const r = dividir(10, 3);
if (r.success) {
  console.log(r.data);   // ✅ TypeScript sabe: .data existe
} else {
  console.log(r.error);  // ✅ TypeScript sabe: .error existe
}
💡
¿Por qué es mejor que try/catch?

Con try/catch, el tipo del error es unknown. Con Result, el tipo del error es explícito y TypeScript lo verifica. Además, quien llame la función no puede olvidarse de manejar el error porque tiene que revisar result.success.

Módulos e import type

TypeScript 2026 con Bun usa módulos ES (import/export). La mejora importante es import type: permite importar tipos puros que se eliminan en compilación, optimizando el bundle final.

typescriptLa forma correcta de importar en 2026
// ─── archivo: usuario.ts ───

export type RolUsuario = "admin" | "editor";
export function crearUsuario(nombre: string): Usuario { ... }

// ─── archivo: app.ts ───

// ❌ import type de todo: crearUsuario no existe en runtime!

Proyecto Integrador — Repositorio Genérico

Esta es la pregunta final del parcial. Combina todos los conceptos: genéricos con restricciones, patrón Result, Map, arrow functions, y interfaces.

typescriptRepositorio genérico — la respuesta correcta completa
// ─── 1. Interfaz base: todo lo que manejamos tiene id y fecha ───
interface EntidadBase {
  readonly id: string;   // readonly: no se modifica después de crear
  createdAt: Date;
}

// ─── 2. Patrón Result: éxito o error sin excepciones ───
type Result =
  | { ok: true; valor: T }
  | { ok: false; error: E };

// ─── 3. Clase genérica: T debe tener las propiedades de EntidadBase ───
class Repositorio {
  // Map: estructura clave-valor para búsqueda eficiente O(1)
  private datos: Map = new Map();

  agregar(entidad: T): Result {
    // Verificar duplicados:
    if (this.datos.has(entidad.id)) {
      return {
        ok: false,
        error: `ID ${entidad.id} ya existe`
      };
    }
    this.datos.set(entidad.id, entidad);
    return { ok: true, valor: entidad };
  }

  // Filtrar con arrow function (callback tipado):
  filtrar(predicado: (e: T) => boolean): T[] {
    return [...this.datos.values()].filter(predicado);
  }

  buscar(id: string): T | undefined {
    return this.datos.get(id);
  }
}

// ─── Uso ───
interface Estudiante extends EntidadBase {
  nombre: string;
  nota: number;
}

const repo = new Repositorio();

const resultado = repo.agregar({
  id: "est-001",
  createdAt: new Date(),
  nombre: "Daniel",
  nota: 4.5
});

if (resultado.ok) {
  console.log("Agregado:", resultado.valor.nombre);
} else {
  console.log("Error:", resultado.error);
}

// Filtrar estudiantes aprobados:
const aprobados = repo.filtrar(e => e.nota >= 3.0);
